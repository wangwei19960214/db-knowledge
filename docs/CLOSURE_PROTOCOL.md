# Closure Protocol / 共享闭环协议

## 定位

本协议是 `business-data-query` 和 `business-data-closed-loop` 两个 skill 共用的唯一共享闭环协议。

两个 skill 都要闭环，但闭环重点不同：

- `business-data-query`：查询型闭环，重点是从用户问题到答案，并判断是否产生可复用查询经验。
- `business-data-closed-loop`：治理型闭环，重点是把一次学习、报表、排查或复盘沉淀成项目知识资产。

其他项目规则、索引、检查清单和 skill 只引用本协议，不复制另一套完整闭环协议。

## 结论等级

所有查询、学习、排查和报表结论必须标明等级：

- `已确认`：已有文档、查询验证、报表校验或业务确认支持。
- `样本验证`：只做了小范围样本、部分对象或部分日期校验。
- `推断`：基于对象名、字段名、历史经验或已有线索推断。
- `待核实`：缺少口径、字段、映射、权限或样本确认。
- `权限不足`：因权限无法确认。
- `仅供排查`：用于定位问题，不作为正式业务口径。

历史文档中的 `已验证` 与本协议的 `已确认` 等价；新增记录优先使用 `已确认`。

## 待核实规则

以下情况必须标注 `待核实`：

- 表名、字段名或指标口径不确定。
- 输入对象类型不确定。
- SKU / Model / 平台 SKU / 上架 SKU / 链接 SKU 映射不确定。
- `campaign_id` / `campaign_name` 对应关系不确定。
- NULL、0、空字符串、缺失记录含义不确定。
- 表粒度、唯一键或去重规则不确定。
- 一对多 join 是否放大不确定。
- FineBI 和 SQL 对账不一致。
- 过程、事件、触发器、刷新时间或血缘不确定。
- 权限不足导致无法确认。

高价值待核实项写入 `docs/business/待核实问题清单.md`。

## 文档路由

按实际相关性更新，不机械更新所有文档。

新增 Markdown 文件不是默认闭环动作；只有现有文档无法承载、用户明确要求，或需要保存不可压缩的历史原文时才考虑新增。能通过压缩、归档、合并入口或更新已有索引解决的，优先不新增文件。

| 情况 | 更新位置 |
| --- | --- |
| 单次需求、执行过程、产物、坑点 | `docs/business/业务流程需求沉淀.md`、`docs/business/需求沉淀索引.md` |
| 新指标、新公式、新口径 | `docs/business/指标口径字典.md` |
| 新表、新字段、新数据集、新关联关系 | `docs/business/数据表目录.md`、`docs/business/数据库对象学习记录.md` |
| 新 FineBI 看板、组件、字段、血缘 | `docs/business/看板到数据集映射.md`、相关 `docs/finebi/` |
| 新校验方式、新对账逻辑 | `docs/business/校验规则.md` |
| 新异常、新坑点、权限问题、字段混用风险 | `docs/business/已知坑点.md` |
| 高价值待核实事项 | `docs/business/待核实问题清单.md` |
| 新取数意图、问法、知识卡片、SQL 模板 | `docs/knowledge/` |
| 新结果表血缘、存储过程映射、刷新任务 | `docs/lineage/` |
| 可复用流程 | `docs/playbooks/` |
| 可复用脚本 | `scripts/` |
| 正式报表产物 | `reports/固定流程取数和报表/` 和 `reports/INDEX.md` |
| 高价值摘要 | `docs/summaries/business-summary.md` |

未读取过程、事件、触发器正文时，不得写成已确认血缘，只能写待核实线索。

## 什么时候需要沉淀

出现以下任一情况，应更新相关文档或转入 `business-data-closed-loop`：

- 新确认字段含义、表粒度、主键、唯一键或映射关系。
- 新确认指标口径、过滤条件、时间口径或汇总规则。
- 新发现字段混用、join 放大、NULL / 0 / 缺失含义风险。
- 新形成 SQL 模板、脚本、报表或可复用流程。
- 新发现 FineBI 差异、血缘线索、刷新规则或权限问题。
- 本次查询或排查形成可复用问法、意图或知识卡片。

## 什么时候不需要更新文档

以下情况可以不更新文档，但最终输出要说明原因：

- 一次性快速问数，且没有新增口径、字段、SQL 模板、坑点或待核实项。
- 结果只用于临时排查，且不形成可复用经验。
- 完全复用已有知识，没有新发现。
- 用户明确要求只查询不沉淀。
- 用户明确进入快速临时取数 / 迭代改表模式，例如“临时取数”“刚刚生成的表改一下”“不要固化”“不要沉淀”“不用更新文档”，且没有新增可复用口径、字段、SQL 模板、坑点或正式报表产物。
- 本次没有生成报表产物，因此不更新 `reports/INDEX.md`。
- 本次没有确认血缘，因此不更新 `docs/lineage/`。

## 快速临时取数 / 迭代改表模式

本模式用于连续临时取数、刚生成文件的快速修改、一次性导出预览、或用户明确要求不固化 / 不沉淀的场景。

执行要求：

- 优先复用已有查询结果、中间 JSON / CSV / Excel 或刚生成的工作簿；没有必要不要重新连接数据库或重跑全量 SQL。
- 只做最小必要校验，并在最终回复说明校验项。
- 本模式生成的 Excel / CSV / JSON / Markdown 默认保存到 `reports/临时取数和报表/<YYYY-MM-DD_任务名>/`；默认只保存结果文件，SQL / 脚本不进入 `scripts/`、不写入索引；如确需复现，可把 `.sql` / `.js` 伴随临时产物放在同目录，但仍不算固定流程。
- 除非用户明确确认固定流程、正式版、固化、沉淀或加入索引，不写入 `reports/固定流程取数和报表/`，不把临时代码提升为 `scripts/` 下的可复用脚本。
- 默认不更新 `reports/INDEX.md`、业务沉淀、知识卡片、血缘文档和 SQL 模板库；最终回复写明未更新文档原因。
- 普通数据 Excel 使用静态轻量生成方式，避免触发 `inspect.ndjson`、`*.xlsx.inspect.ndjson`、大型预览日志或重型视觉检查。
- 如果过程中产生新的可复用口径、字段含义、坑点、SQL 模板、脚本或正式报表产物，应在最终回复标注为“待沉淀”；用户确认正式版、固化或沉淀后，再按本协议更新相关文档，并把可复用脚本放入 `scripts/`。

## 用户回复展示规则

Closure Packet 是闭环字段标准，不要求每次面向用户原样展开所有空字段。

- 面向用户的最终回复可以压缩展示，只保留本次有信息量的字段。
- 写入报告、交接文档或治理记录时，应保留完整字段，空项可写“无”或“本次未涉及”。
- 不能因为压缩展示而省略关键风险、待核实项、权限不足、产物路径或是否更新文档。
- 如果用户要求“完整闭环”或任务涉及复杂报表 / 文档治理，应优先展示完整 Closure Packet。

## 两个 skill 的协作

- `business-data-query` 负责从用户具体问题到业务答案。
- `business-data-query` 如果发现新知识、新口径、新坑点、新 SQL 模板或新报表产物，应输出 Closure Packet，并转入或建议使用 `business-data-closed-loop`。
- `business-data-closed-loop` 负责将查询、学习、报表和排查结果沉淀为项目知识资产。
- 两个 skill 都必须遵守只读数据库、安全凭据保护、按相关性读取文档、不编造口径和字段的规则。

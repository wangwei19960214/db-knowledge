# Markdown 文档健康检查

生成日期：2026-06-18

> 状态说明（2026-06-30）：本文件是 2026-06-18 的历史健康检查报告，部分问题已在后续治理中修复；当前入口和路径以 `docs/INDEX.md`、`docs/governance/文档治理瘦身与闭环收敛执行工单.md` 和 `reports/INDEX.md` 为准。以下正文保留原审计结论，不作为最新全量状态。

## 范围

- 检查对象：项目自有 Markdown 文档，不含 `node_modules` 和 `.git`。
- 覆盖文件：检查输入 26 个；本审计报告为新增产物，不计入输入清单。
- 约计行数：7,316 行。
- 包含目录：`AGENTS.md`、`docs/`、`reports/`、`.agents/skills/business-data-closed-loop/`。

## 总体结论

当前文档没有发现明显乱码、Unicode 替换字符、异常控制字符或未闭合代码块；也没有在 Markdown 中发现数据库密码、token、api key 等敏感凭据。

更明显的问题集中在文档治理和口径同步：

1. 旧路径和新路径同时存在，尤其是 `AGENTS.md` 前半段仍引用根目录旧文档名。
2. 多份文档已经追加 2026-06-18 内容，但检查时文件头 `更新日期` 仍停留在 2026-06-16 或 2026-06-17。
3. 广告利润口径在不同 summary / playbook 中同步不一致，容易让后续 agent 误判“总成本/总利润”是否仍完全待核实。
4. 部分大文档已经超过 1,000 行，摘要和原文之间需要更明确的同步规则。
5. `reports/` 中测试产物、失败产物和正式报表混放，后续自动检索时需要区分状态。

## 明显坑点

### 1. `AGENTS.md` 旧路径会误导启动流程

- 位置：`AGENTS.md`
- 现象：文档分工和默认阅读顺序仍写 `数据库数据梳理.md`、`业务流程需求沉淀.md`、`FineBI库存数据梳理.md` 等根目录文件。
- 实际：根目录不存在这些文件，当前真实路径是：
  - `docs/business/数据表目录.md`
  - `docs/business/业务流程需求沉淀.md`
  - `docs/finebi/FineBI库存数据梳理.md`
- 风险：后续 agent 如果只按前半段说明找文件，会找不到或误以为文档缺失。
- 建议：把 `AGENTS.md` 前半段旧路径统一改成 `docs/` 新路径；保留“旧路径已迁移”的一句说明即可。

### 2. 多个文件头 `更新日期` 已过期

- `docs/business/数据表目录.md`：文件头为 2026-06-16，但正文已有多条 2026-06-18 更新。
- `docs/business/业务流程需求沉淀.md`：检查时文件头为 2026-06-17，但正文已有 2026-06-18 需求记录；本次追加闭环记录时已同步改为 2026-06-18。
- `docs/finebi/FineBI广告数据梳理.md`：文件头为 2026-06-17，但正文已有大量 2026-06-18 内容。
- `docs/finebi/FineBI库存数据梳理.md`：文件头为 2026-06-16，文件实际修改时间为 2026-06-17。
- 风险：如果后续按文件头判断最新程度，会错过最新口径。
- 建议：每次沉淀时同步更新文件头，或者统一改成“最近结构性更新 / 最近追加记录”两种日期。

### 3. 广告利润口径状态在不同文档中不同步

- 最新较完整状态见：
  - `docs/summaries/business-summary.md`
  - `docs/finebi/FineBI广告数据梳理.md`
  - `docs/business/指标口径字典.md`
  - `docs/business/校验规则.md`
- 已确认部分：
  - `WF店铺绩效2.0/广告` 的 `总成本`、`总利润` 公式层已确认。
  - DDP、数量、出货费、单买数量等右侧来源已有 2026-06-18 补充确认。
- 仍待核实部分：
  - `每个sku每月平均成本` 的未显示上游。
  - 历史 `平均ddp修` 与本轮 `ddp平均成本` 的命名关系。
  - `上架sku利润` 预览 0 行原因。
- 同步风险：
  - `docs/summaries/finebi-ad-summary.md`、`docs/playbooks/广告组月报生成流程.md`、`docs/summaries/report-playbooks-summary.md` 仍偏向写“总成本/总利润上游生成待核实”。
- 建议统一表述为：`总成本/总利润公式层与关键右侧来源已确认；每个sku每月平均成本、平均ddp修/ddp平均成本命名关系、上架sku利润 0 行原因仍待核实。`

### 4. 标题结构有自动目录风险

- `docs/business/数据表目录.md` 有两个一级标题：
  - `# 数据库数据梳理`
  - `# 第一版业务语义层目录`
- `.agents/skills/business-data-closed-loop/references/update-templates.md` 的第二个一级标题在代码块模板中，不影响渲染，但会干扰简单 `rg '^#'` 或自动目录脚本。
- `docs/business/业务流程需求沉淀.md` 的模板代码块内有标题，也会被简单标题扫描误识别。
- `docs/finebi/FineBI广告数据梳理.md` 中 `### 3.2` 出现两次：
  - `王威 / 新广告：新广告明细与 attribution 数据集`
  - `王威 / 增量更新：WF 广告明细链路`
- 建议：
  - `数据表目录.md` 第二个一级标题改为 `## 第一版业务语义层目录`。
  - `FineBI广告数据梳理.md` 后半段编号顺延，例如把第二个 `3.2` 改为 `3.3`，后续 `3.3` 到 `3.6` 顺延。
  - 自动审计脚本扫描标题时忽略代码块。

### 5. 大文档过长，摘要同步压力越来越大

- `docs/finebi/FineBI库存数据梳理.md`：1,067 行。
- `docs/finebi/FineBI广告数据梳理.md`：1,018 行。
- `docs/business/数据表目录.md`：887 行。
- `docs/business/业务流程需求沉淀.md`：861 行。
- 风险：后续 agent 虽然按 summary 读取，但 summary 如果滞后，会把大文档里的最新事实藏起来。
- 建议：
  - 给每个大文档加“最新状态”短章节，限制 10 到 20 行。
  - summary 文件只保留当前可执行口径和待核实清单，不重复大量历史。
  - 对广告、库存大文档按主题拆分或至少增加目录锚点。

### 6. `reports/` 产物状态混放

- 当前 `reports/` 下有：
  - `晨间简报测试输出_2026-06-12.md`
  - `类目Top20断货晨报_2026-06-17_失败.md`
  - 本审计报告
- 风险：后续汇总报告产物时，测试/失败产物可能被当成正式交付。
- 建议：
  - 增加 `reports/INDEX.md` 记录每个产物的状态：正式、测试、失败、审计。
  - 或拆分目录：`reports/final/`、`reports/test/`、`reports/failed/`、`reports/audit/`。

## 可优化点

### 路径和链接

- 当前多数路径是反引号内联文本，不是 Markdown 链接。
- 优点：简单。
- 缺点：无法自动做 broken link 检查。
- 建议：关键入口文档，例如 `docs/INDEX.md`、summary、playbook，可逐步改成 Markdown 相对链接。

### `待核实` 数量偏多

只读扫描中，`待核实` 出现较多的文件包括：

- `docs/business/数据表目录.md`：约 99 处。
- `docs/business/指标口径字典.md`：约 43 处。
- `docs/business/业务流程需求沉淀.md`：约 25 处。
- `docs/business/看板到数据集映射.md`：约 23 处。
- `docs/finebi/FineBI广告数据梳理.md`：约 17 处。

这本身符合“未知必须标注”的规则，但需要按优先级沉淀成 backlog，否则会变成噪声。建议新增一个 `docs/business/待核实事项清单.md` 或在 `docs/INDEX.md` 增加“高优先级待核实”入口。

2026-06-29 治理复核：当前已使用 `docs/business/待核实问题清单.md` 作为高价值待核实入口，并补充生命周期索引；后续不再新增 `待核实事项清单.md`，避免形成第二套待核实入口。

### 长行较多

部分公式说明行超过 420 字，主要在广告成本、利润和右侧来源描述中。

风险：

- 人工阅读费劲。
- diff 审核困难。
- 后续小修改容易形成大段冲突。

建议把长行拆成“公式、来源、匹配键、限制”四个 bullet。

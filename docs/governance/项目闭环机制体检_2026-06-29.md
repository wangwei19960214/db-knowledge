## 总体判断

项目闭环已经从“单次任务沉淀”升级成了较完整的知识治理体系：有共享闭环协议、两类 skill 分工、待核实池、需求索引、知识层、血缘层、报表索引和检查清单。

当前主要问题不再是“没有闭环”，而是“闭环入口太多、索引有漂移、沉淀文档增长过快”。如果不治理，后续 agent 会越来越依赖摘要和索引，但摘要 / 索引一旦滞后，就会比原始文档更危险。

4. `docs/knowledge/` 和 `docs/lineage/` 已把智能取数知识与血缘信息从业务流水账中分离出来。

## 当前闭环问题

### 1. 闭环协议唯一，但周边文件仍有重复规则

`AGENTS.md`、`docs/INDEX.md`、`docs/CLOSED_LOOP_CHECKLIST.md`、两个 skill、summary 和 reference 模板都在描述启动顺序、文档路由、最终回复或沉淀要求。

风险：

- 后续规则更新时容易只改一个入口，其他入口变成旧口径。
- agent 在执行时不知道该以 `AGENTS.md`、skill 还是 `CLOSURE_PROTOCOL.md` 的描述为准。

建议：

- 保留 `docs/CLOSURE_PROTOCOL.md` 作为唯一完整规则。
- 其他文件只保留短引用和入口差异，不再复制完整路由。
- `docs/CLOSED_LOOP_CHECKLIST.md` 的“最终回复应包含”建议直接写“按本次入口 skill 输出对应 Closure Packet，可压缩展示”。

### 2. 默认读取顺序存在轻微分叉

- `AGENTS.md` 的默认阅读顺序包含 `docs/summaries/business-summary.md`。
- `business-data-closed-loop` 的 Required Reading 不包含 business summary。
- `docs/INDEX.md` 默认只读写法与后面“学习沉淀闭环必读”写法不完全一致。

风险：

- 有的 agent 会读 summary，有的不会，导致是否看到最新摘要取决于入口解释。

建议：

- 统一为：所有数据库 / FineBI / 报表 / 智能取数任务默认读 `AGENTS.md`、`docs/INDEX.md`、入口 skill、`docs/CLOSURE_PROTOCOL.md`；是否读 `docs/summaries/business-summary.md` 按任务复杂度设为“推荐读”，不要在多个地方写成不同必读。

### 3. 完整 Closure Packet 有价值，但可能过重

`business-data-closed-loop` 要求最终回复必须包含完整 Closure Packet。对大型报表和学习任务很好；但对这类文档治理问答，完整字段会显得机械。

风险：

- agent 为了填满字段，产生空泛内容。
- 用户只想看结论时，最终回复被 Closure Packet 挤占。

建议：

- 在协议里增加“压缩 Closure Packet”规则：面向用户的最终回复可压缩到 5 到 8 行，但报告或沉淀文档中保留完整版。
- 对“无 SQL / 无指标 / 无业务范围”的治理任务，允许字段写 `不适用`，但不要省略结论等级、产物路径、待核实和文档更新状态。

### 4. 报表索引和磁盘实际文件不一致

检查发现 `reports/INDEX.md` 当前索引了不存在的路径：

- `reports/固定流程取数和报表/REQ-005_历史晨报/类目Top20断货晨报_2026-06-17_失败.md`
- `reports/固定流程取数和报表/REQ-005_历史晨报/类目Top20断货晨报_2026-06-23_失败.md`
- `reports/固定流程取数和报表/REQ-005_历史晨报/类目Top20断货晨报_2026-06-24_失败.md`
- `reports/固定流程取数和报表/REQ-011_Wayfair广告组分月报表/finebi_competitor_ui_mockup_2026-06-23/`

同时磁盘上存在但索引未单列的产物或辅助文件，例如：

- `reports/固定流程取数和报表/SUPPORT_数据库对象地图/database_catalog_enriched_2026-06-25.xlsx`
- `reports/固定流程取数和报表/REQ-004_Wayfair有货率报表/wayfair_availability_2026-01_to_2026-06-15/Wayfair_上架SKU及链接SKU有货率_2026-01_至_2026-06-15_复刻版.xlsx`
- `reports/固定流程取数和报表/REQ-004_Wayfair有货率报表/wayfair_availability_2026-01_to_2026-06-15/Wayfair_上架SKU及链接SKU有货率_2026-01_至_2026-06-15_美化版.xlsx`

风险：

- 后续按索引打开失败产物会踩空。
- 漏索引的增强版产物可能被重复生成。

建议：

- 给 `reports/INDEX.md` 增加 `路径存在` 或 `最近核验日期` 字段。
- 对不存在但有历史意义的记录，状态改为 `历史记录-文件缺失`。
- 对增强版数据库目录和多版本有货率报表，明确“正式版 / 复刻版 / 美化版 / 辅助数据”。

### 5. 需求沉淀索引没有覆盖最新治理类需求

`docs/business/需求沉淀索引.md` 已有 15 条，但文档治理和闭环体检类记录没有持续进入索引。

风险：

- 后续要找“闭环系统为什么这样设计”“哪里查治理问题”时，必须全文搜 `业务流程需求沉淀.md`。

建议：

- 所有治理 / 体系建设 / 审计类任务也进入需求索引。
- 给治理类需求单独用 `REQ-GOV-*` 或继续顺延 `REQ-016`，避免被业务报表淹没。

### 6. 文件头更新日期滞后

扫描到正文已有更晚日期，但文件头仍旧的文件：

- `docs/business/业务流程需求沉淀.md`：文件头 2026-06-18，正文已有 2026-06-29。
- `docs/business/数据表目录.md`：文件头 2026-06-19，正文已有 2026-06-29。
- `docs/finebi/FineBI公共数据目录地图.md`：文件头 2026-06-17，正文已有 2026-06-25。
- `docs/finebi/FineBI广告数据梳理.md`：文件头 2026-06-22，正文已有 2026-06-26。
- `docs/finebi/FineBI库存数据梳理.md`：文件头 2026-06-16，正文已有 2026-06-26。

风险：

- agent 或人工按文件头判断新旧时，会错过最新记录。

建议：

- 文件头改成两字段：`创建/结构版本日期` 和 `最近追加日期`。
- 每次闭环追加时，只更新 `最近追加日期`，避免纠结历史章节的原始日期。

### 7. 大文档继续膨胀，流水账压力明显

当前较大文件：

- `docs/business/业务流程需求沉淀.md`：2,354 行。
- `docs/business/数据表目录.md`：1,315 行。
- `docs/finebi/FineBI广告数据梳理.md`：1,306 行。
- `docs/finebi/FineBI库存数据梳理.md`：1,170 行。
- `docs/knowledge/SQL模板库.md`：896 行。

风险：

- “索引优先”越来越依赖摘要质量。
- 同一主题的新发现分散在流水账、数据表目录、FineBI 专项、knowledge 卡片和 lineage 中，查找成本升高。

建议：

- `业务流程需求沉淀.md` 保留近 30 天或近 20 条，历史按月份归档到 `docs/business/archive/`。
- `数据表目录.md` 按主题拆成 `inventory.md`、`advertising.md`、`sales.md` 等，主文件只做目录。
- `FineBI广告数据梳理.md` 按看板 / 主题拆分，保留一个总入口。

### 8. 高价值待核实池很好，但需要生命周期

`docs/business/待核实问题清单.md` 已经按编号收敛问题，但部分线索已经变化，例如 `Q-AUTO-001` 仍写“失败文件存在”，而对应失败 Markdown 当前并不在磁盘。

风险：

- 待核实项变成长期挂账，状态不可信。

建议：

- 增加列：`首次发现日期`、`最近复核日期`、`负责人/下一动作`、`关闭条件`。
- 状态从自由文本收敛为：`待核实`、`部分确认`、`已确认`、`已关闭`、`废弃/文件缺失`。

### 10. 临时文件规则已建立，但工作区仍有治理残留

风险：

建议：

- 对 `workbook_validation.ndjson` 做二选一：保留并改名为小型验证结果，或迁移为 `.json`。

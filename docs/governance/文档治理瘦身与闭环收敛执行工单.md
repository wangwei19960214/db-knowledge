## 执行边界

允许做：

- 更新 Markdown 索引、入口说明、文件头日期和治理记录。
- 给不存在但有历史意义的索引路径追加“文件缺失 / 待确认”状态。
- 把已确认存在的报表产物补进 `reports/INDEX.md`。
- 把超大流水文档的历史内容迁移到 `archive/`，但必须保留原文；默认复用现有归档入口，不为入口再建新文件。
- 为待核实项增加生命周期字段或状态说明。
- 追加冲突说明，不能静默覆盖历史结论。

禁止做：

- 连接数据库、执行 SQL、打开 FineBI 或调用会改变数据的流程。
- 删除历史业务记录、报表产物、脚本或附件。
- 为了“看起来更清楚”新增索引、README、总览或临时说明文档。
- 把 `待核实` 直接改成 `已确认`，除非有明确证据。
- 把已确认业务口径改写成新口径。
- 改 `.env`、凭据、数据库连接配置。
- 默认清理 `scripts/node_modules`、`.DS_Store`、sidecar 文件；清理前必须确认它们不是交付产物或必要运行时。

## Phase 4：待核实池生命周期

目标：让 `docs/business/待核实问题清单.md` 不变成长期挂账池。

建议增加字段：

- `首次发现日期`
- `最近复核日期`
- `负责人 / 下一动作`
- `关闭条件`

建议状态：

- `待核实`
- `部分确认`
- `已确认`
- `已关闭`
- `废弃 / 文件缺失`

不要把旧问题机械关闭。缺少证据时保留 `待核实`，只更新下一动作。

维护后必须校验：

- 生命周期索引和问题明细的 `Q-` 编号数量一致。
- 生命周期索引没有比问题明细多出或少掉的编号。
- 生命周期索引保持 5 列，问题明细保持 8 列。
- `关闭条件` 只代表未来可关闭标准，不代表当前已确认；状态仍以问题明细为准。
- `最近复核日期` 必须带复核类型；结构复核不能当作业务确认，且不要因结构维护批量刷新未触达问题的日期。

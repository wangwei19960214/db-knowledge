# 数据库数据梳理

最近追加日期：2026-07-03

## 范围

- 数据库：`zzdnb`
- 方向：库存、销售、断货、售罄、采购、物流、财务、客诉、广告、FineBI 报表等业务数据。
- 用途：作为长期数据库业务地图，记录核心表、字段口径、上下游关系和更新逻辑。

## 使用原则

- 按业务主题组织，不按字母顺序逐表罗列。
- 优先记录已经被需求、脚本、FineBI 看板验证过的表和口径。
- 对未验证内容明确标注“待核实”，避免把推断写成事实。
- 新表、新口径、新业务链路在这里沉淀；单次需求过程写入 `业务流程需求沉淀.md`。

#### `总账单详情_extract`

- 类型：实体表，InnoDB。
- 当前数据范围：2026-04-01 至 2026-06-26；2026-06-26 15:50 只读校验时共 `53,681` 行。
- 2026-07-01 复核：当前共 `57,889` 行，覆盖 `2026-04-01` 至 `2026-07-01`；`3` 个平台、`12` 个店铺、`4,717` 个上架 SKU、`1,741` 个采购 SKU，物理主键重复数 `0`。
- 业务粒度：物理主键为 `上架sku + 出单店铺 + SKU + PO`，一行表示某平台店铺某 PO、某上架 SKU 拆出的一个采购 SKU / 组件行。Wayfair 场景下，`上架sku` 已是订单拆分后的子件 part number。
- 核心字段：`出单平台`、`出单店铺`、`PO`、`订单日期`、`上架sku`、`SKU`、`数量`、`售价`、`sku总价`、`订单总价`、`订单来源`、`上架包含采购种类`、`发货仓库/发货仓库细分`、`订单状态`、`取消数量`。
- 数据来源：`wayfair_orders_allinfo`、`wayfair_orders_allinfo_api`、`鲸汇wayfair订单管理`、`overstock_orders_allinfo`、`homedepot_orders_allinfo`，并通过对应平台映射拆到采购 SKU。2026-06-26 用户业务确认：`overstock上架sku销售映射`、`homedepot上架sku销售映射` 是组合件对应采购 SKU；2026-06-29 样本修正：`wayfair上架sku销售映射.part_number` 是店铺 SKU / Supplier Part Number，可能是 `Kit`、`Standard` 或 Component，并映射到采购 SKU；用户口头称“销量映射表”，数据库真实表名为“销售映射”。
- 刷新链路：事件 `总账单详情_extract_wayfair` 每 1 小时调用过程；`总账单详情_extract_overstock`、`总账单详情_extract_homedepot` 每 2 小时调用过程。过程会写入/更新本表，并维护 `总账单发货仓库映射`、`上架sku销售映射缺失记录` 等辅助对象。
- 触发器：`zzdxq_insert_copy2_copy1` 在插入前写 `create_time=now()`；`zzdxq_update_copy2_copy1` 在更新前写 `update_time=now()`。
- 金额口径：组合上架 SKU 会拆成多采购 SKU 行；`售价 * 数量` 是当前可用于采购 SKU / 组件层的分摊金额。`sku总价` 是上架 SKU 总价，在多组件行上会重复；`订单总价` 也会在同一 PO 下重复，不能直接按行求和。
- 组合产品销量 / 有货率建议：做组件库存或销量权重时可用 `SKU` 关联采购 SKU 库存；2026-06-29 用户业务确认 `数量` 是销售个数，`取消数量` 必须扣减。做上架 SKU 或订单金额时应先按 `出单店铺 + PO + 上架sku` 去重。
- 2026-07-01 取消数量补充：有取消记录的订单上架 SKU 中，Overstock 有 `4` 个多组件订单上架 SKU、Wayfair 有 `2` 个多组件订单上架 SKU 出现取消数量按组件行重复；有效销量不能直接逐行 `数量 - 取消数量`，应先回到订单上架 SKU 粒度确认取消件数，再按组件数量扣回。
- 2026-07-03 取消数量补充：2026-06 样本中 `Wayfair / Cancelled` 行 `183`、PO `157`、`取消数量=807`；`Overstock / CANCELLED` 行 `44`、PO `38`、`取消数量=47`；Homedepot 2026-06 样本取消数量为 `0`。按店铺看，取消数量主要来自 `WF-TD=501`、`WF-FOP=162`、`WF-TD-CA=111`、`WF-TD-UK=48`、`OS-SP=23`、`OS-FOP=15`、`OS-LJD=9`。
- 与 `wf上架sku每日销售额` 关系：`wf上架sku每日销售额.supplier_part` 是销售源 Supplier Part Number / 销售 SKU 层，`总账单详情_extract.上架sku` 是订单拆分子件层；二者不能直接等值 join 当作同粒度。若 `supplier_part` 对应 `Kit` 或父件，需要先用 `wfproduct_management_latest.Related_Kit_Part -> Supplier_Part_Number` 展开到子件，再接采购 SKU。
- 待核实：`上架包含采购种类` 早期 NULL 行是否可完全用实际 `COUNT(DISTINCT SKU)` 替代；OS / HD 取消数量扣减 SQL 已做 2026-06 样本验证，Wayfair 销售额分摊到子件 / 采购 SKU 的正式 SQL 仍待核实。

### 选表与应用规则

- 问某 SKU 每日预测销量：按场景查 `每日销量预估`、`_amz` 或 `_垂直`，不要查物理表 `销量预估` 代替每日输入。

- 问当前仓库层实际入库：查 `仓库入库单` 的 `_CG` 仓库分支；问 AMZ / 垂直累计拆分：对 `cg入库产品拆分_修改` 按 `柜号 + Part + 仓库名称` 求和。
- 问店铺分到的入库量：查 `店铺库存拆分入库数量_cg`，但必须先对账同键上游累计量，并输出陈旧 / 不守恒异常。
- 当前正数已完成 CG 入库的 15,586 个业务键均能在产品拆分表命中且数量一致；产品拆分表另有 271 个历史或辅助来源键，不能仅因未命中当前仓库正数事实就删除。
- 主映射路径 17,847 个相关源行中有 11 行命中两个采购 SKU，结果表也出现 11 个多采购 SKU 候选组；组合件是否允许一对多需按 Part 逐项确认。

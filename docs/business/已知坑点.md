# 已知坑点

## 使用原则

- 新坑点追加记录，不覆盖旧记录。
- 未确认根因标注“待核实”；推断根因标注“推断”；权限问题标注“权限不足”。
- 坑点必须尽量写清影响指标、影响范围和规避方式。

## WSC 与 retail sales 不能混用

- 发现日期：2026-06
- 业务场景：广告销售额、ROAS、广告销售占比、广告利润分析。
- 影响范围：WF-WSP 广告看板、wf广告、广告组月报。
- 涉及表 / 数据集 / 看板：`advertising_product_report_by_day`、`wf广告`、`WF广告数据/广告`、`WF-WSP广告组广告分析`。
- 补充涉及数据集：公共数据 `王威/新广告` 下 Sponsored Shops / Google Shopping 数据集，例如 `zzdnb_sponsored_shops_sku_level_attribution`、`zzdnb_google_shopping_sku_level_attribution`、`zzdnb_google_shopping_enhanced_attribution`。
- 涉及指标：广告销售额、ROAS、总销售额、广告销售占比。
- 现象：广告看板既能看到 WSC / wholesale cost，也能看到 retail sales 字段；新广告数据集中还出现 `WSC Revenue`、`Wholesale Revenue`、`Sales` 等不同命名。
- 根因：已确认 WF 广告利润口径中的广告销售额取 WSC 字段。
- 风险：ROAS、广告销售额、利润分析口径混乱。
- 规避方式：字段名和报表说明必须写明 WSC / WSC Revenue / Wholesale Revenue / Sales / retail sales 的具体来源；WF-WSP 广告利润默认使用 WSC。新广告字段级等价关系未核实前，不默认互相替代。
- 校验方式：同时抽查 `attributed_wholesale_cost_window_view_through_USD_Day_14` 和 `attributed_retail_sales_window_view_through_USD_Day_14`。
- 状态：已确认
- 相关需求记录：FineBI 广告利润口径梳理。

## 广告事实状态字段不要默认硬过滤

- 发现日期：2026-07-07
- 业务场景：Wayfair 广告表现、WSP 广告组月报、ROAS / CTR / CVR / CPC 等指标取数。
- 影响范围：`advertising_product_report_by_day`、`wf广告`、`WF广告数据/广告`、`WF-WSP广告组广告分析`。
- 涉及字段：`isB2b`、`campaign_status`、`campaign_is_active`、`product_status`。
- 现象：2026-01 至 2026-05 中，`isB2b=true` 有 `63,480` 行、`spend=30,600.4100`、WSC `515,552.1500`；非 `ACTIVE/True/ACTIVE` 状态组合也存在非零花费和 WSC。
- 根因：广告事实表保留平台原始状态和 B2B 拆行；状态字段表示广告 / 商品状态，不等同于“是否应从历史表现里剔除”。
- 风险：默认 `WHERE isB2b='false'` 或只保留 `campaign_status='ACTIVE' AND product_status='ACTIVE'` 会低估历史花费、WSC、订单和件数，进而影响 ROAS、CTR、CVR、CPC 等汇总指标。
- 规避方式：默认广告表现查询保留这些状态字段，作为维度或异常说明输出；只有用户明确要求、或复刻具体 FineBI 组件过滤时，才按指定状态过滤。
- 校验方式：按状态过滤前后对比 `SUM(spend_USD)`、WSC、orders、units；汇总层比率按过滤后的分子分母重算。
- 状态：数据库侧已确认；FineBI 具体组件是否额外过滤仍待核实。
- 相关需求记录：2026-07-07 Wayfair 广告事实状态过滤边界学习。

## 最新 / 近 14 天广告 Day_14 归因未成熟

- 发现日期：2026-07-07
- 业务场景：Wayfair 近期广告表现、最近几天 ROAS、当日 / 昨日广告销售额、广告订单和件数查询。
- 影响范围：`advertising_product_report_by_day`、`WF店铺绩效2.0/广告`、`WF广告数据/广告`、WSP 广告组报表。
- 涉及字段：`date_est`、`spend_USD`、`attributed_wholesale_cost_window_view_through_USD_Day_14`、`attributed_orders_window_view_through_Day_14`、`attributed_units_window_view_through_Day_14`、FineBI `是否满足14天`。
- 现象：2026-07-07 只读样本中，最新业务日 `2026-07-06` 有 `1,095` 行和花费 `1,255.6988`，但 WSC、retail、orders、units 均为 `0`；`2026-07-05` 已有非零 WSC `24,508.6610`。
- 根因：样本验证 + FineBI 线索。广告销售额、订单和件数字段是 Day_14 归因字段；FineBI 处理链存在 `是否满足14天 = IF(DATEDIF(date_est,NOW(),"D")>14,"已满足","未满足")`。
- 风险：把最新日或近 14 天 WSC / ROAS 当作最终口径，会误判广告效果；把最新日 WSC 为 0 误判为整表未刷新，也会导致错误排查方向。
- 规避方式：近期问数默认同时输出成熟度说明；正式趋势、月报或 ROI / ROAS 结论默认排除未满足 14 天日期，或单独展示未成熟区间。
- 校验方式：按 `date_est` 聚合 spend、WSC、orders、units，并结合 FineBI `是否满足14天` 判断成熟度；最新日期需同时看是否有行数和 spend。
- 状态：样本验证；具体 FineBI 组件是否默认过滤 `是否满足14天` 仍待核实。
- 相关需求记录：2026-07-07 Wayfair 广告 Day_14 归因成熟度学习。

## 新广告表不能直接按全表或展示字段名取数

- 发现日期：2026-07-08
- 业务场景：`wf新广告表现`、WF 广告汇总 WGS / WSD / WSS / WSV、Google Shopping、Sponsored Display、Sponsored Shops、Sponsored Video 分析。
- 影响范围：新广告广告销售额、ROAS、CTR、订单、花费、利润辅助字段。
- 涉及表 / 数据集 / 看板：`google_shopping`、`google_shopping_sku_level_attribution`、`google_shopping_enhanced_attribution`、`google_shopping_attribution_new_to_wayfair`、`sponsored_display_*`、`sponsored_shops_*`、`sponsored_video_*`、FineBI `wf新广告表现`。
- 现象：FineBI 数据集名常带 `zzdnb_` 前缀，但当前数据库物理表通常无前缀；FineBI 展示字段常用空格名，物理字段多为下划线名。Sponsored 系列表存在多批 `create_time`，直接全表汇总会重复；`sponsored_display_sku_level_attribution` 最新批次内也存在 `store + 年月 + Class + SKU` 重复。
- 根因：样本验证；FineBI 抽取数据集、物理表和展示字段之间存在命名层，且抽取批次未确认是否清空重建。
- 风险：SQL 照抄 FineBI 数据集名会找不到表；直接全表求和会放大花费、订单和 WSC；直接用展示字段名会报字段不存在或取错字段。
- 规避方式：先用物理表名和物理字段名；Sponsored 系列表默认取最新 `create_time` 批次，再按目标业务粒度聚合；比率字段按分子分母重算，不直接平均。
- 校验方式：查 `information_schema.columns` 确认物理字段名；查 `MAX(create_time)` 与候选键重复；对 WSC / Retail / Wholesale 做汇总对账。
- 状态：样本验证；FineBI 组件是否已过滤最新批次仍待核实。
- 相关需求记录：2026-07-08 Wayfair 新广告表物理字段和业务粒度学习。

## campaign_name 可能变更

- 发现日期：2026-06
- 业务场景：广告组月报、BID 关联、广告组趋势分析。
- 影响范围：按广告组汇总或关联出价表的报表。
- 涉及表 / 数据集 / 看板：`advertising_product_report_by_day`、`zzdnb_wf广告出价表`、`广告组合单买详情`、`WF-WSP广告组广告分析`。
- 涉及指标：广告花费、广告销售额、ROI、ROAS、BID。
- 现象：广告事实表有 `campaign_id` 和 `campaign_name`；出价表有 `Campaign name`，同时也有 `campaignlayer_id`。
- 根因：广告组名称可能变更或重名；2026-06-18 数据库抽样发现同一 `campaignlayer_id` 存在多个 `Campaign name` 样本。
- 风险：按名称 join 会错配历史数据或丢失出价。
- 规避方式：主分析优先使用 `campaign_id`；补当前 bid 快照时优先使用 `wf广告出价表.campaignlayer_id` 对 `advertising_product_report_by_day.campaign_id`，`Campaign name` 只作展示和兜底。只有名称时保留未匹配清单并标注名称风险。
- 校验方式：检查同一 ID 多名称、同一名称多 ID、出价表名称未匹配、`campaignlayer_id` 未匹配。
- 2026-07-01 补充：`wf广告出价表` 主键为 `store + Campaign name + wayfair SKU + campaignlayer_id + 日期`；2026-01 至 2026-05 出价表 `100,892` 行且 `campaignlayer_id` 无空值。2026-05-31 抽样出现多条广告事实按 `campaignlayer_id` 可匹配但按 `Campaign name` 不匹配的记录；当时覆盖率未收口，正式报表建议预聚合后输出未匹配清单。
- 2026-07-07 补充：2026-01 至 2026-05 覆盖率复核中，广告事实 `135,395` 行按 `campaignlayer_id = campaign_id` 匹配 `124,503` 行，按 `Campaign name` 匹配 `121,437` 行，仅名称匹配 `0` 行。`WF-TD-CA`、`WF-TD-UK` 在 `wf广告出价表` 中未见店铺快照，是未匹配主因之一；不能未经业务确认直接拿 `WF-TD` BID 回填 CA / UK。
- 状态：已确认名称风险；`campaignlayer_id` 是当前优先匹配键，但仍需输出未匹配清单，CA / UK 出价兜底规则待核实。
- 相关需求记录：Wayfair 2026年1-5月广告组库存综合分析。

## WSP 广告组 `bid` 字段名易混淆

- 发现日期：2026-06-18
- 业务场景：`WF-WSP广告组广告分析`、`WSP广告组每月汇总`、`WSP广告组明细表`、广告组 bid 趋势和出价分析。
- 影响范围：广告组月报、按 bid 分组分析、当前出价核对。
- 涉及表 / 数据集 / 看板：`advertising_product_report_by_day`、`WF广告数据/广告`、`WSP广告组每月汇总`、`WSP广告组明细表`、`zzdnb_wf广告出价表`。
- 涉及指标/字段：`bid`、`product_max_bid`、`Default bid(现在的bid)`、`Suggested Bid detail(具体金额)`、`最小建议bid`、`最大建议bid`。
- 现象：`WSP广告组每月汇总` 组件维度槽显示短字段 `bid`，容易被误认为出价快照表的 `Default bid(现在的bid)`。
- 根因：2026-06-18 FineBI 组件字段信息已确认，月汇总组件里的 `bid` 来源字段名是 `product_max_bid`、来源表是 `广告`；`WSP广告组明细表` 另外直接展示 `Default bid(现在的bid)`、建议 bid 等出价表补充字段。
- 风险：把 `product_max_bid` 当作当前最新 bid，或把建议 bid 当作当前 bid，会导致出价效果分析和当前出价核对口径错误。
- 规避方式：按问题选择字段：历史广告效果按 bid 分组时使用 `advertising_product_report_by_day.product_max_bid`；核对“当前最新 bid”时使用出价快照表最新日期的 `Default bid(现在的bid)`；核对历史出价时使用 `store + date_est/日期 + campaign_name/Campaign name + sku/wayfair SKU` 的日期匹配字段。
- 校验方式：在 FineBI 组件中点击字段查看 `来源字段名`；在广告底表中按 `store + 月份 + campaign_id/name + sku + product_max_bid` 分组，检查同一 campaign/SKU 是否因不同 bid 拆成多行。
- 2026-07-01 补充：2026-01 至 2026-05 广告事实中，`3,828` 个 `store + month + campaign_id + sku` 组里有 `718` 组存在多个 `product_max_bid`，最多 `5` 个；这不是异常，而是历史 bid 效果分析需要保留的维度。
- 状态：已确认
- 相关需求记录：2026-06-18 Wayfair WSP 广告组报表 `bid/当前bid` 字段来源核实。

## WSP 广告组总销售额按 bid 拆行后会重复

- 发现日期：2026-06-23
- 业务场景：最终 WSP 广告组数据表、`WF-WSP广告组广告分析 -> 广告组活动 -> WSP广告组每月汇总`。
- 影响范围：总销售额、广告销售占比、按广告组 / SKU / bid 输出的月报。
- 涉及表 / 数据集 / 看板：`WF广告数据/广告`、`公共数据/王威/广告/zzdnb_wf上架sku每日销售额`、物理表 `wf上架sku每日销售额`、`WSP广告组每月汇总`。
- 涉及字段：`总销售额1`、`总销售额修`、`销售额`、`total_revenue`、`wayfair_sku`、`sku`、`product_max_bid`。
- 现象：`WSP广告组每月汇总` 的总销售额相关字段来自 SKU 月销售额 `wf上架sku每日销售额.total_revenue`，按 `store + 月份 + sku=wayfair_sku` 补到广告事实。若最终结果继续按 `bid(product_max_bid)`、日期或多行广告事实拆分，同一个 SKU 月销售额会出现在多行。
- 根因：`wf上架sku每日销售额` 的粒度是 `store + 日期/月 + wayfair_sku`，不是 `campaign` 或 `bid` 原生事实；FineBI 通过 `AVG_AGG(总销售额1)` / `总销售额修` 处理展示分母。
- 风险：直接对拆分后的 `总销售额1` 求和，会放大总销售额，并进一步压低或扭曲广告销售占比。
- 规避方式：最终表可以把该销售额对到广告组和月份，但向上汇总前必须按唯一 `store + 月份 + campaign + sku` 去重，或复用 FineBI `AVG_AGG(总销售额1)`；不要用 `wf每月产品和广告数据` / `wf每月产品和广告数据汇总` 替代此组件的总销售额来源。
- 校验方式：按 `store + 月份 + sku` 汇总 `wf上架sku每日销售额.total_revenue`，与接入后按同键去重的销售额对齐；再检查按 bid 行直接求和与去重后求和的差异。
- 状态：已确认来源和风险
- 相关需求记录：2026-06-23 WSP 广告组每月汇总总销售额来源修正。

- 业务场景：组合产品销量有货率、Wayfair 总销售额分摊、采购 SKU 组件销量、广告总销售额与订单拆分对账。

### Wayfair WSP 广告组剩余缺口核实：标签、价格、总销售额、利润可抽取性

- 提出时间：2026-06-19
- 业务目标：只核实 `活动的垂直标签`、`是否有价格优势` 物理来源、FineBI 精确 `总成本/总利润` 是否可直接抽取、`总销售额1` 来源、原 `断货百分比` 字段来源，不重复核实 WSC、bid、广告订单、评分评论和有货率。
- 执行方式：

  - 使用只读 Node / MySQL 查询 `INFORMATION_SCHEMA`、候选物理表字段和样本；每次查询后关闭连接。
- 已核实表 / 数据集：
  - `wf上架sku每日销售额`
  - `wf_supplier_parts_performance`
  - `产品近两天的最小前台价`

- 关键结论：
  - 数据库未发现物理列 `活动的垂直标签`、`广告标签修`；高可信物理候选为 `上架sku利润表.垂直标签/垂直细分标签` 经 `wfproduct_management.SKU` 桥接到广告 SKU 后，在 FineBI 中拼接/重命名为 `活动的垂直标签`。该字段按月变化，样本中存在 `在售,新品`、`下架,清仓,滞销` 等多标签拼接形态。
  - `本竞品价格对比` 未发现同名物理表；可用 `产品近两天的最小前台价` 复刻 `本品最低前台价/竞品最低前台价`。复刻键为本品 `store + link_sku + model`，竞品按同 `platform + model` 聚合最小 `最近日期最小前台价`；回补广告仍按 FineBI 已见 `Display_SKUs = link_sku`、`store = 店铺`。该表是当前快照，不是月度历史。
  - `总成本/总利润` 未发现可直接抽取的广告精确物理列；同名字段命中在非广告精确利润链路表。当前可落地方案是预聚合 DDP、发货费、单买数量、广告事实和 SKU 映射后按 FineBI 公式复算；不能用 `上架sku利润表` 冒充广告精确利润。
  - `总销售额1` 的 `销售额` 来自 FineBI 同源物理表 `wf上架sku每日销售额.total_revenue`，按 `store + LEFT(日期,7) + wayfair_sku` 回补。2026-01 至 2026-05 与 `wf_supplier_parts_performance` 对比，美国店 `WF-FOP/WF-TD` 完全一致；`WF-TD-CA/WF-TD-UK` 行数一致但金额不同，涉及 CA/UK 时应优先使用 FineBI 同源表。
  - 原 `断货百分比` 链路已确认到 `WF店铺绩效2.0/wf广告`：从 `各sku每月断货天数` 补 `可用`，再生成 `上架sku可用库存`、`断货part个数`、`活动的上架sku数量`，组件公式为 `SUM_AGG(断货part个数) / SUM_AGG(活动的上架sku数量)`。
- SQL / 查询注意：
  - 尝试大范围三表桥接 `advertising_product_report_by_day -> wfproduct_management -> 上架sku利润表` 超过 60 秒，已主动中断；后续应使用单 SKU / 单月样本或预聚合小表，不要直接扫大范围。
  - 查询期间曾出现一次连接建立阶段 `ETIMEDOUT 101.37.199.205:3306`，随后 `SELECT 1` 成功恢复；记录为网络波动。

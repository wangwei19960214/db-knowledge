# 待核实问题清单

本清单只收高价值问题，不机械搬运所有出现“待核实”的文字。优先记录会影响广告、库存、有货率、利润、销售、FineBI 对账、自动化稳定性的事项。

## 生命周期维护规则

- `首次发现日期`：旧问题如果无法从当前摘要准确追溯，不回填臆测日期；后续只能依据归档原文、查询记录或业务确认补充，无法追溯则继续保留 `待补`。
- `最近复核日期`：必须标明复核类型，例如 `YYYY-MM-DD（结构复核）` 或 `YYYY-MM-DD（业务复核）`。2026-06-29 初建生命周期索引时仅做结构复核，不代表业务内容已确认。
- 后续不要因为结构维护而批量更新所有 `最近复核日期`；只更新实际被结构复核、业务复核或状态处理过的编号。
- `负责人 / 下一动作`：优先写可执行验证方式；没有明确负责人时写“待业务 / 后续 agent 按主题复核”。
- `关闭条件`：必须有文档、只读查询、FineBI 样本或业务确认支撑，才能从 `待核实` / `部分确认` 改为 `已确认` 或 `已关闭`。
- `关闭条件` 是未来可关闭标准，不代表当前已满足；当前结论仍以“问题明细”的 `状态` 为准。
- 下方“生命周期索引”只记录治理元数据，不改变问题状态或业务口径；逐条业务复核完成后，再按证据更新原问题表。
- 新增、合并或关闭问题明细时，必须同步更新生命周期索引；编号以“问题明细”为准，避免索引漂移。
- 每次维护后必须做结构校验：生命周期索引与问题明细的编号数量一致、无缺失 / 额外编号，生命周期索引保持 5 列，问题明细保持 8 列。

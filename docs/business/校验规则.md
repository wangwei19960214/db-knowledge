# 校验规则

## 使用原则

- 报表交付前必须做时间范围、粒度、重复、映射、关键指标和异常清单检查。
- 比率指标必须按分子分母重算，不能直接平均明细比率。
- 对账时优先使用已确认 FineBI 看板或历史报表；无法对账时标注“待核实”或“权限不足”。

## 广告月度总量对账

- 适用场景：WF 广告组月报、整体广告分析、分类广告分析。
- 相关指标：广告花费、点击、曝光、广告销售额、广告订单数、广告件数、自然销售额、总销售额。
- 相关表 / 数据集：`advertising_product_report_by_day`、`wf广告`、`WF广告汇总`、`总广告数据`、`分类目广告数据`。
- 校验目标：将明细 SQL 按月/店铺汇总后，与 FineBI `WF-WSP整体广告分析` 或 `wf广告` 看板核心指标对齐。
- 校验 SQL / 脚本：待沉淀；已有经验记录在 `docs/business/业务流程需求沉淀.md` 的 Wayfair 广告组分析。
- 通过标准：月份、店铺、核心指标总量一致或差异在业务可接受阈值内；阈值待用户确认。
- 异常处理：输出差异清单，包含月份、店铺、指标、SQL 值、FineBI 值、差异额、差异率。
- 最近验证日期：待核实
- 相关需求记录：Wayfair 2026年1-5月广告组库存综合分析。

## 日期覆盖检查

- 适用场景：所有 SQL、报表、FineBI 对账。
- 相关指标：全部时间序列指标。
- 相关表 / 数据集：广告日报、库存快照、销量表、断货事实表、报表产物。
- 校验目标：确认每个数据源覆盖用户要求的开始日期和结束日期。
- 校验 SQL / 脚本：按表输出 `MIN(date)`、`MAX(date)`、日期数、缺失日期数；具体脚本待沉淀。
- 通过标准：核心事实表覆盖完整；如广告、销量、库存截止日期不同，必须在报表中分别注明。
- 异常处理：输出数据源、期望开始/结束、实际开始/结束、缺失日期、影响指标。
- 最近验证日期：2026-06 Wayfair 有货率报表已记录库存、销量、广告截止日期不同。
- 相关需求记录：Wayfair 上架 SKU 与链接 SKU 有货率。

## Wayfair 有货率映射预检

- 适用场景：Wayfair 上架 SKU 有货率、链接 SKU 有货率、Model 有货率、广告库存综合分析。
- 相关指标：上架 SKU 有货率、链接 SKU 有货率、断货天数、缺映射数量、异常清单。
- 相关表 / 数据集：`平台产品详情表`、`wayfair上架sku销售映射`、`采购sku_model映射表`、`仓库映射表`、`warehouse_stock_在售`。
- 校验目标：正式计算有货率前，确认上架范围、专用组件映射、回退采购 SKU、无效链接 SKU、国家仓映射不会导致缺失或放大。
- 校验 SQL / 脚本：可复用 `docs/knowledge/SQL模板库.md` 中 `SQL-WF-AVAILABILITY-MAPPING-PRECHECK`。
- 必查项：
  - `平台产品详情表` 中 Wayfair `store + 上架sku` 是否唯一。
  - `wayfair上架sku销售映射` 是否按 `store + part_number + 采购sku` 唯一，且 `part_number` 只按 `上架sku` 关联。
  - 上架行命中专用映射、可回退 `平台产品详情表.采购sku`、无采购 SKU 可回退的数量。
  - 回退 `采购sku` 是否含逗号多值，含逗号时必须拆分。
  - `链接sku` 是否为 `NULL`、空字符串或 `无`，链接层聚合前必须过滤。
  - `仓库映射表` 关联前是否按 `后台仓库名称 + 所在国家` 去重，并排除 `后台仓库名称 = '无'`。
- 通过标准：有货率主表能区分专用映射、回退映射和缺映射；所有缺映射、无效链接、仓库歧义均进入异常清单或备注。
- 异常处理：不要静默跨店铺借用 `part_number`；不要把 `链接sku='无'` 合并为有效链接；不要直接 join 未去重的 `仓库映射表`。
- 最近验证日期：2026-06-26 Wayfair 有货率映射闭环。
- 相关需求记录：Wayfair 上架 SKU 与链接 SKU 有货率。

## 三平台上架 SKU 销售映射预检

- 适用场景：组合产品销量有货率、`总账单详情_extract` 订单拆分、三平台 part number 到采购 SKU 映射、缺映射排查。
- 相关指标：组件销量、采购 SKU 销量、组合产品有货率权重、缺映射 key、组件数量。
- 相关表 / 数据集：`wayfair上架sku销售映射`、`overstock上架sku销售映射`、`homedepot上架sku销售映射`、`总账单详情_extract`、`上架sku销售映射缺失记录`。
- 校验目标：正式计算前确认真实表名、平台层级、主键唯一性、组件数分布、`数量` 字段质量和订单抽取映射覆盖率。
- 校验 SQL / 脚本：可复用 `docs/knowledge/SQL模板库.md` 中 `SQL-PLATFORM-LISTING-PURCHASE-MAP-PRECHECK`。
- 必查项：
  - 三张表是否按 `store + part_number + 采购sku` 唯一。
  - `part_number`、`采购sku`、`数量` 是否为空，`数量` 是否有非数字。
  - 每个平台 `store + part_number` 的组件数分布和最大组件数。
  - `总账单详情_extract` 最近业务范围内按 `出单店铺 + 上架sku + SKU` 是否能命中对应平台映射。
  - Wayfair 场景必须确认 `wf上架sku每日销售额.supplier_part` 是销售源 Supplier Part Number / 销售 SKU 层，`总账单详情_extract.上架sku` 是订单子件层，不能直接等值当作同层级；如果要做组合件分摊，还要先识别 `Product_Type`。
  - Wayfair `WF-TD-CA`、`WF-TD-UK` 当前由 `WF-TD` 映射派生，涉及 CA / UK 对账时必须单独标注。
- 通过标准：输出平台级覆盖率、缺映射 key、组件数分布和待核实项；任何缺映射或跨层级 join 都必须进入异常清单或结论限制。
- 异常处理：不要静默丢弃缺映射订单；不要把业务口头“销量映射表”误写成不存在的物理表；不要把 `数量` 误当组件个数，正式有效销量必须扣减 `取消数量`；Wayfair 组合件销售额分摊前必须先按 Component 行展开到子件并输出缺映射异常。
- 最近验证日期：2026-06-26 三平台上架 SKU 销售映射表学习。
- 相关需求记录：组合产品销量有货率 - `总账单详情_extract` 与三平台映射表学习。

## CG 仓库存预检

- 适用场景：Wayfair CG 仓库存、后台 CG 可用、CG 仓在途、FineBI `cg仓库存` 复刻或对账。
- 相关指标：CG 仓可用、CG 仓在途、实际标品库存、未映射 part 数、未映射可用库存。
- 相关表 / 数据集：`wayfair后台cg可用`、`cg_part_sku映射`、`warehouse_stock_最新`、FineBI `cg仓库存`。
- 校验目标：正式取 CG 仓库存前，确认最新日期、源表粒度、状态分布、映射覆盖和采购 SKU 展开是否会放大库存。
- 校验 SQL / 脚本：可复用 `docs/knowledge/SQL模板库.md` 中 `SQL-WF-CG-INVENTORY-PRECHECK`。
- 必查项：
  - `wayfair后台cg可用` 最新 `create_time` 日期和最大 `update_time`。
  - 最新日期按 `store + part_number + warehouse` 是否唯一。
  - `ACTIVE / INACTIVE` 行数和可用库存分布。
  - `cg_part_sku映射` 是否存在一对多组件映射。
  - 源表左连映射后的行数是否大于源表行数。
  - 未映射 part 行数、未映射可用库存和未映射在途。
  - 与 `warehouse_stock_最新` CG 仓汇总差异是否可解释。
- 如果目标是对齐 `warehouse_stock_最新`，额外校验 `ACTIVE + 已映射 + 排除 90670 源仓` 后的可用，以及 `ACTIVE + 已映射 on_order` 的在途。
- 通过标准：输出粒度明确；映射展开不误作为库存总量；未映射、状态过滤、90670 源仓归类规则清晰。
- 异常处理：不要静默丢弃未映射 part；不要在未确认 `INACTIVE` 处理前只看 `ACTIVE`；不要把 `wayfair后台cg可用` 和 `warehouse_stock_最新` 的 CG 仓数直接互相替代。
- 最近验证日期：2026-06-26 CG 仓库存专项 / CG 仓库存对账收口。
- 相关需求记录：库存主题精学 / CG 仓库存专项 / CG 仓库存对账收口。

## WSP 广告报表粒度与映射放大检查

- 适用场景：Wayfair WSP 广告月报、广告组报表、广告效果结合链接 SKU / 有货率 / 链接利润率。
- 相关指标：广告花费、广告销售额(WSC)、广告订单数、广告件数、点击、曝光、CPC、CTR、CVR、CPA、ROAS、ROI、链接利润率。
- 相关表 / 数据集：`advertising_product_report_by_day`、`WF广告数据/广告`、`公共数据/王威/增量更新/wf广告`、`cg_part_sku映射时间范围`、`wayfair上架sku销售映射`、`平台产品详情表`、`zzdnb_鲸汇店铺销售产品`。
- 校验目标：
  - 确认 `bid` 使用广告底表 `product_max_bid`，且作为维度时同月同 campaign/SKU 的不同 bid 会拆行。
  - 确认 WSP `广告订单数` 使用 `attributed_orders_window_view_through_Day_14` / 汇总层 `orders`。
  - 确认广告指标在广告事实粒度先聚合，再补链接 SKU、有货率、链接利润率，避免采购 SKU 展开导致指标放大。
  - 确认最终表总销售额使用 `zzdnb_wf上架sku每日销售额 / wf上架sku每日销售额.total_revenue`，按 `store + 月份 + sku=wayfair_sku` 接入；接入后按 bid 或日期拆行时不能直接累加 `总销售额1`。
- 推荐广告聚合键：`store + 年月 + campaign_id + campaign_name + sku + product_max_bid`；如果不看 bid 效果，需先明确业务口径后去掉 `product_max_bid`。
- 推荐重算公式：`CPC = SUM(spend_USD) / SUM(clicks)`；`CTR = SUM(clicks) / SUM(impressions)`；`CVR = SUM(attributed_orders_window_view_through_Day_14) / SUM(clicks)`；`CPA = SUM(spend_USD) / SUM(attributed_orders_window_view_through_Day_14)`；`ROAS = SUM(WSC) / SUM(spend_USD)`。
- 总销售额校验：先按 `store + 月份 + wayfair_sku` 汇总 `wf上架sku每日销售额.total_revenue`，再与最终表中按 `store + 月份 + sku` 去重后的 `总销售额1` 对账。若最终表包含 `campaign`，应明确该 SKU 月销售额是被带到广告组维度的经营分母；若同 SKU 同月多个 campaign 均展示该销售额，跨 campaign 汇总时需按业务口径决定是否去重或保留重复展示，不能把它当 campaign 原生销售额。
- 必查异常清单：
  - 广告事实键重复 / 同一广告键多 bid。
  - `first_10_part_numbers` 拆分后一个广告 SKU 对多个上架 SKU。
  - 上架 SKU 对多个链接 SKU 或多个采购 SKU。
  - 无有效链接 SKU、无采购 SKU、无数量、无有货率、无链接利润率。
  - 展开后广告花费、WSC、点击、曝光、订单数与聚合前不一致。
  - 按 bid / 日期拆行后 `总销售额1` 直接求和大于按 `store + 月份 + sku` 去重后的销售额。
- 已验证抽样基准：
  - 单一映射：`WF-TD / 2026-01 / campaign_id=342884 / sku=OARI1049 / bid=0.3 / first_10_part_numbers=TD-A-41907LED` -> `上架SKU=TD-A-41907LED / 链接SKU=W110467359 / 采购SKU=41907LED x 1`，不应改变广告指标总量。
  - 多映射：`WF-TD / 2026-03 / campaign_id=347274 / sku=OARI1014 / bid=0.3` -> `7` 个上架 SKU、`14` 个采购 SKU 组件、同一 `链接SKU=W110073209`；展开后如直接求和会放大广告指标。
  - 多 bid：`WF-FOP / 2026-05 / campaign_id=458816 / sku=UXIF3679` 存在 `product_max_bid=0.4/0.6/0.8` 三组；按 bid 维度应分别汇总广告效果。
- 查询实现注意：历史映射表存在 `utf8mb3/utf8mb4` collation 混用，抽样或校验 SQL 如遇 collation 报错，可显式 `CONVERT(字段 USING utf8mb4)` 后比较；不要因为查询报错改业务 join 键。
- 通过标准：聚合前后广告指标总额一致；所有一对多映射均进入异常清单或有明确分摊规则。
- 异常处理：不直接删除重复；输出异常 sheet / JSON，并在主报表中保留映射来源和是否分摊标记。
- 最近验证日期：2026-06-18
- 相关需求记录：Wayfair WSP 广告报表字段来源和实现粒度核实。

## WSP 广告标签来源检查

- 适用场景：`WF-WSP广告组广告分析` 中按 `清仓/滞销/在售/新品/下架` 拆分组件，或报表输出 `标签` / `活动的垂直标签`。
- 相关字段：`广告细分标签`、`垂直标签`、`广告标签`、`活动的垂直标签`、`广告标签修`。
- 相关表 / 数据集：`WF店铺绩效2.0/广告`、`WF广告数据/广告`、`zzdnb_公司sku状态表`。
- 校验目标：确认 `活动的垂直标签` 是否由 `zzdnb_公司sku状态表` 补入的 `广告细分标签/垂直标签` 生成，而不是误用数据库同名物理列或直接取 `上架sku利润表` 明细。
- 校验步骤：
  - 在 FineBI 上游 `WF店铺绩效2.0/广告` 中确认添加列节点：`UPPER(采购SKU)=UPPER(sku)`，`公司标签/model/广告细分标签/垂直标签` 聚合方式为最大值。
  - 如需要输出 model，同时确认 `zzdnb_采购sku_model映射表` 添加列节点：`UPPER(采购SKU)=UPPER(采购sku)`，`model` 聚合方式为最大值；不要把这条 model 补充线误当作标签来源。
  - 确认 `广告标签` 公式：`新品1-3 -> 新品3个月`，`新品4/5/6 -> 新品6个月`，`新品7 -> 在售`，否则取 `垂直标签`。
  - 确认 `活动的垂直标签 = 按 [extend,isB2b] 组内字符串拼接 [广告标签]`。
  - 确认 `广告标签修` 再按 `清仓 -> 滞销 -> 在售 -> 新品6个月 -> 新品3个月 -> 下架 -> 空` 的包含优先级归类。
- 通过标准：能说明来源表、匹配键、拼接粒度和最终归类优先级；数据库没有同名物理列时不得强行写成物理字段。2026-06-22 已确认 `zzdnb_公司sku状态表` 对应物理表 `公司sku状态表`，FineBI 当前使用当前快照，无月份 join。
- 异常处理：若同一 `[extend,isB2b]` 内拼接出多个标签，按 `广告标签修` 优先级归一；若 `zzdnb_公司sku状态表` 无匹配，输出空标签 / 待核实清单。若报表要求历史月度标签，应另接 `公司sku状态表_记录留存` 或 `公司sku状态表_temp_2_记录留存` 的 `留存date`，不要直接把当前快照当 2026-01 至 2026-05 历史标签。2026-06-23 覆盖核实：留存表从 2026-03 起才有快照，2026-01/02 必须标注历史快照缺失或使用当前快照兜底。
- 最近验证日期：2026-06-23
- 相关需求记录：FineBI WSP 广告组剩余口径闭环。

## WSP 广告分类分区检查

- 适用场景：`WF-WSP分类广告分析`、`WF-WSP广告组广告分析`、广告组报表按 `室内/户外/灯具/其他` 分区输出。
- 相关指标：广告花费、广告销售额(WSC)、点击、曝光、订单数、利润、ROI、ROAS、分类自然销售额、分类总销售额。

## CG 入库事实到店铺库存拆分一致性校验

- 适用场景：SPO 到仓、CG 实际入库、AMZ / 垂直拆分、店铺入库分配和店铺 FIFO 排查。
- 业务键：跨仓库事实、产品拆分和店铺拆分对账必须使用 `入库单号 + 柜号 + Part + 仓库名称`；店铺明细再加 `store`。物理主键缺少 `入库单号`，不能代替完整业务键。采购 SKU 作为映射结果单独做一对多检查。
- 源级检查：`wayfaircg入库` 先按 `store + Inbound_Order_ID + Part_Number` 检查计划行唯一性；`wayfaircg入库时间` 的数量字段先排除空 / `--` 并验证可转数值。
- 映射检查：按过程实际日期规则连接 `cg_part_sku映射时间范围`，输出未命中和命中多个采购 SKU 的源行；多命中不得任取一行。
- 仓库事实检查：分别统计已完成 / 待入库、缺入库时间、待入库但已有实际量、`柜号 + Part + 仓库` 多采购 SKU。
- 产品拆分通过标准：按业务键汇总 `cg入库产品拆分_修改` 全部增量行后，累计实际量与 `仓库入库单` 已完成正数实际量一致；每行 `AMZ入库数量 + 垂直入库数量 = 实际入库数量`。负修正行保留参与汇总。
- 店铺拆分通过标准：先过滤现行生产范围 `状态='已完成'`，再按完整业务键汇总 `店铺入库数量` 等于产品拆分累计实际量；目标表保存的实际量与上游一致；源键全部覆盖。另按过程实际 `柜号 + Part` 存在性键检查是否被其他入库单历史记录误拦截。差异必须输出多分 / 少分、差额、首次下游时间和最新上游时间。
- 调度检查：验证 `:10` 仓库入库、`:19` CG 产品拆分、`:20` 店铺分配源、`:20` CG 店铺总控的结果时间；同刻任务不能仅凭 `LAST_EXECUTED` 推断先后。
- 历史快照：2026-07-15 13 点复核为 45 个不守恒、223 个保存实际量陈旧；原“1 个缺失”包含不进入生产过程的待处理状态，不作为已完成源键缺失基线。
